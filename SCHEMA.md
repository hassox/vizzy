# SCHEMA.md — Vizzy Schema-Driven Contract

This repository is **schema-driven**. Data shapes are decided here, not in ad-hoc TypeScript. Read this before changing event or config types.

## Where shapes live

- **Registry (authoritative):** [Rusl](https://rusl.com) — schemas and bundles are versioned, reviewed, and published there.
- **Vendored snapshots:** `./schemas/` — generated by `rusl install` from `rusl.bundle.toml`. Never edit these by hand (except local drafts under `./schemas/dan/` before publish).

### `dan/schemas/geo-event` (locked for v1)

One message = one full journey. Visual phases (launch → flight → impact) are owned by the viz, not the wire format.

| Field | Required | Notes |
|---|---|---|
| `id` | yes | Unique event ID (UUID) |
| `correlationId` | no | Groups related journeys (maps to OTel `traceId` via adapter) |
| `origin` | yes | GeoJSON Point (`pragmatic/schemas/geo` `$defs/point`) |
| `destination` | yes | GeoJSON Point |
| `magnitude` | no | 0–100 intrinsic scale (blast yield, packet size, tonnage). Default 50 |
| `priority` | no | 0–100 urgency / attention relative to other events. Default 50 |
| `velocity` | no | 0–100 relative travel speed (display scale, not SI). Default 50 |
| `metadata` | no | Theme-specific extras; unknown keys ignored |
| `emittedAt` | yes | ISO 8601 when the producer emitted the event |

**Rejected fields:** `type` (theme owns rendering), `severity` (use magnitude + priority), `stages` (viz-derived lifecycle).

**Semantics:**

- `magnitude` — how big the event is in absolute terms
- `priority` — how much attention it demands relative to peers
- `velocity` — relative travel speed; theme maps it to arc duration

All 0–100 fields are **display scales**, not physical units.

### `dan/schemas/viz-config` (local draft)

**One schema** for client configuration. Root document is what we save (localStorage / export). Reusable pieces live in `$defs` — not separate Rusl resources.

```
viz-config (root = library envelope)
  $defs.source         named feed bookmark
  $defs.sourceBinding  how a run connects
  $defs.preset         full run / wall configuration
```

**Root (library / export file):**

| Field | Required | Notes |
|---|---|---|
| `version` | yes | `1` |
| `sources` | yes | Array of `$defs/source` (max 100) |
| `presets` | yes | Array of `$defs/preset` (max 50) |
| `activePresetId` | no | Last applied preset |
| `updatedAt` | no | ISO 8601 |

**`$defs/source`** — bookmark only (`id`, `name`, `url`, optional notes/timestamps).

**`$defs/preset`** — one run:

| Field | Required | Notes |
|---|---|---|
| `id` | yes | Stable id |
| `version` | yes | Preset doc version; currently `1` |
| `title` | yes | Wall masthead **and** picker label (no separate name) |
| `description` | no | Wall about text |
| `theme` | no | Default `ember` |
| `maxConcurrent` | no | 10–500; default 200 |
| `audio` / `spin` | no | Defaults false / true |
| `playbook` | no | Generator scenario id or null |
| `source` | yes | `$defs/sourceBinding` |

**`$defs/sourceBinding` modes:**

| `mode` | Required | Behavior |
|---|---|---|
| `url` | `url` | Ad-hoc WebSocket |
| `saved` | `sourceId` | Resolve against `sources[]` |
| `cycle` | `sourceIds` | Rotate library sources every `cycleIntervalSec` (default 60, min 5) |

Presets reference sources by id (no embedded copies) so renaming a feed does not fork presets. `mode=url` is the one-off escape hatch.

**Not a wire format.** Only `geo-event` rides the WebSocket.

## Local-first development

New schemas are drafted as local JSON Schema files under `./schemas/dan/` and iterated on before publishing to Rusl. Once a schema is stable:

1. Propose it on Rusl under the `dan` account via CLI or MCP tools.
2. Review and publish.
3. Replace the local draft with a `rusl add dan/schemas/<slug>` dependency.
4. Run `rusl install` to vendor the published version.

## How to refresh vendored schemas

```sh
rusl install          # re-resolve from rusl.bundle.toml + rusl.lock
rusl add <schema-id>  # add a new dependency
rusl outdated         # check for newer versions
```

## How to change a shape

1. Propose or update the schema on Rusl (via CLI `rusl` or MCP tools), or edit the local draft under `./schemas/dan/` while iterating.
2. Review and publish the new version on the registry (when ready).
3. Update `rusl.bundle.toml` with the new version constraint.
4. Run `rusl install` to vendor the updated schema.
5. Regenerate any code (e.g., `zodforge generate` for TypeScript).
6. Update consuming code to match.

## Meaning layer

Schemas define shape. Meaning (PII flags, retention policy, domain interpretation, what a `magnitude: 80` means in context) lives as Rusl annotations attached to schemas. Before writing code that depends on a field's meaning, check for annotations on the schema version.

## Iron rule

Before creating, modifying, or generating code for any data type, you must read this file first.
